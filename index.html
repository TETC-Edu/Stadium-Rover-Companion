Level 1 — Blocks

One delivery. One cargo, one zone, one stack of blocks. Make it work.

Blocks
Objective

Make the rover deliver one cargo to its zone using VEXcode Blocks.

What you're doing

Pick one delivery from your stadium — just one. Measure the distance from the supply room to the zone with the ruler. Measure the turn angle with the protractor. Translate those numbers into a stack of blocks. Run it. Watch what happens.

You're not trying to be perfect. You're trying to get the rover to do a thing.

This is the inventory, not a recipe. The order and the numbers are yours.

Stuck? Ask a partner. Then ask me.

Open one card at a time. Each card has three hints. We give you the next one when you ask.

?The rover isn't moving the right distance.
Try first Stop. Measure again. Make sure your ruler and your block agree on units. VEX moves in millimeters. 1 cm = 10 mm.
Then this Run it once and watch where it actually stops. Is it short, long, or in the right place but pointed wrong? That tells you which number to change.
Approach Distance and angle are two separate problems. Fix one at a time. Try a clean drive forward (angle = 0) first, then come back and adjust the heading.
?The rover is turning the wrong way.
Try first Look at the protractor on the field. Which direction is 0°? Which way is positive? Sketch it before you change anything.
Then this VEX measures heading like a compass: 0 is forward, positive turns right, negative turns left. The sign of your number matters as much as the size.
Approach Where is the rover pointed now? Where does it need to point? Say it out loud with your partner. The turn is the difference between those two — speak it before you type it.
?The kicker isn't firing.
Try first Is the Kicker block actually in your stack? It's easy to drop a Drive block where you meant a Kicker. Look closely.
Then this The kicker has two modes: kick and place. Soccer ball gets kicked. Barrels get placed gently. Are you using the right one?
Approach Order matters. The rover has to be at the zone before the kicker fires. If it fires mid-drive, the cargo lands wherever the rover happens to be. Walk through your stack, one block at a time.

Same delivery, new layout. Convert your blocks to Switch, then edit the Python values.

Switch
Objective

Convert your Level 1 stack to a Switch block. Read the Python it generates. Edit the numbers to match your new stadium.

What you're doing

Match 2 is in a different city. The stadium moves. The supply room, the zones, the distances — at least two of them are in a new place. Your job is to keep the same delivery working in the new world.

In VEXcode, right-click your block stack → Convert stack to Switch block. Your blocks become Python inside a single Switch container. You don't have to write Python yet. You just have to find the numbers and change them.

Level 3 — Text Python

          
Knockout round. Three matches, three days, no time to rewrite the code.

        

        Python
      


      

        Objective
        
All three deliveries. One list. One loop. Numbers at the top — the loop does the rest.

      


      

        
What you're doing

        
The stadium moves every match. Your code can't. So instead of typing the same drive-place-return sequence three times, you write it once and feed it three deliveries.

        
Start a new text Python project in VEXcode AIM. You can drag and drop Python blocks, or type from scratch — typing is encouraged. The starter below shows the shape.


        

          Why this matters —
          In Level 2 you edited four numbers in four places. In Level 3, you change the numbers in one place. The loop does the rest. That's the whole reason Python exists.
        

      


      

        
Starter scaffold

        
Copy this into your project. Fill in the blanks with your measurements.

        

          

            
          


# My deliveries: (distance_mm, angle_deg, cargo)
deliveries = [
    (___, ___, "soccer_ball"),
    (___, ___, "blue_barrel"),
    (___, ___, "orange_barrel"),
]

for distance, angle, cargo in deliveries:
    robot.screen.print(cargo)
    robot.move_for(distance, angle)
    # drop off the right way for this cargo
    robot.kicker.place()
    # come back to the supply room
    robot.move_for(distance, angle + 180)

Stuck?

        


          

            

              
?I don't know how to start a text project.

              
            

            

              

                Try first
                Open VEXcode AIM. New project → Python (not Blocks). Save it before you write anything.
              

              

                Then this
                Type the line that wakes the robot up first. Run it. If the robot responds, you're ready. If it doesn't, fix that before writing more code.
              

              

                Approach
                Copy the starter scaffold above. Fill in one delivery's numbers. Get that one delivery working before you fill in the other two.
              

              

            

          


          

            

              
?My loop only runs once.

              
            

            

              

                Try first
                Look at indentation. Everything inside the for must be indented the same. Lines that aren't indented are outside the loop.
              

              

                Then this
                Count your delivery entries. A loop runs exactly as many times as there are items in the list. If your list has one item, you get one run.
              

              

                Approach
                Print something inside the loop — robot.screen.print(cargo) — so you can see how many times it runs. The screen is your debugger.
              

              

            

          


          

            

              
?The variables aren't doing what I expected.

              
            

            

              

                Try first
                Say out loud what you expect the variables to do. Then say what they're actually doing. The gap is where the bug lives.
              

              

                Then this
                Variables in the list are unpacked in order: distance, angle, cargo. If you wrote them in a different order, the rover is using your distance as an angle.
              

              

                Approach
                Pick one delivery. Print all three of its values to the screen before the rover moves. If the numbers on the screen are wrong, your list is wrong. If they're right, the bug is somewhere else.

Bonus — Vision Final Approach

          
The stadium just got rearranged five minutes before kickoff. Your code doesn't know the new layout. Your rover does have eyes.

        

        Vision
      


      

        
Locked

        
Finish Level 3 first. The Bonus opens when you mark it complete.

      


      

        

          Objective
          
Use the AI Vision Sensor to find at least one zone instead of hardcoding its position.

        


        

          
The scenario

          
A partner team rebuilt your stadium without telling you what changed. Your delivery list is wrong. Hardcoded distances will miss every time.

          
But the rover can see. The camera reports what color blobs it's looking at right now — their bearing, their width, where they are in the frame. Stop driving by memory. Start driving by what's in front of you.

        


        

          
Vision commands

          


# Look for a specific color
data = robot.vision.get_data(BLUE_BARREL)

# Each item in `data` has:
#   .bearing   how far off-center, in degrees
#   .width     bigger = closer
#   .centerX   horizontal pixel position

if len(data) > 0:
    target = data[0]
    robot.turn_for(RIGHT, target.bearing)
    robot.move_for(100, 0)

          

        


        

          
Stuck?

          


            

              

                
?How do I know what the camera sees?

                
              

              

                

                  Try first
                  Open the camera view on the rover. It shows you which color blobs it's detecting right now.
                

                

                  Then this
                  In code, robot.vision.get_data(BLUE_BARREL) returns information about every blue barrel in view — bearing, width, and position.
                

                

                  Approach
                  Don't drive by vision yet. Stand still and print what the camera sees first. Once you trust the report, then drive.
                

                

              

            


            

              

                
?I see the object but the rover misses it.

                
              

              

                

                  Try first
                  Bearing tells you which way to turn. Width tells you how close you are. You need both.
                

                

                  Then this
                  Width grows as you approach. Decide what width counts as "close enough," then drive until width is bigger than that.
                

                

                  Approach
                  Vision is a feedback loop, not a one-shot calculation. Look, turn, look again, drive, look again. That's the structure.
